## [Unreleased]

## [1.9.1] ‚Äî 2026-03-11

### Removed
- **Cloudflare Worker (`workers/`) eliminated** ‚Äî The entire `autocr-api.nicx.me` / `autocr-cdn.nicx.me` Worker infrastructure is removed. All useful functions absorbed into the static Astro website on Cloudflare Pages (`autocr.nicx.me`):
  - `updates.xml` is now a static file in `website/public/`, generated by `sync-constants.sh`
  - `releases.astro` fetches GitHub Releases at build time instead of proxying through the Worker API at runtime
  - `VersionBadge.astro` reads version from `version.json` at build time instead of fetching from the Worker API
  - `_headers` CSP no longer allows external `connect-src` (no runtime API calls)
  - `version.json` simplified: `domains.cdn` and `domains.api` removed
  - `deploy.yml` no longer has a `deploy-worker` job
  - `.github/workflows/deploy-worker.yml` manual workflow deleted
  - `scripts/check-version.sh` updated (51‚Üí56 checks, all Worker references removed, release-publish workflow guards added)
  - `scripts/sync-constants.sh` updated (Worker syncs removed, `updates.xml` generation added)
  - All URL references updated: `autocr-cdn.nicx.me` ‚Üí `autocr.nicx.me` in installer scripts, Go config, and docs

### Changed
- **Update URL domain** ‚Äî `updateUrl` in `version.json` and all consumer files now points to `https://autocr.nicx.me/updates.xml` (was `https://autocr-cdn.nicx.me/updates.xml`)

### Fixed
- **Canonical release reconciliation** ‚Äî release/public/version surfaces that had drifted back to `1.8.1` now realign to the canonical `1.9.1` declared in `version.json`, including extension/website manifests, installer metadata, generated Pages artifacts, and the root/extension release docs
- **Tag-release website deploy ordering** ‚Äî `.github/workflows/deploy.yml` now makes the release-tag website deployment wait for `create-release`, preventing Pages from publishing download/update links before the GitHub Release assets exist
- **Master website publish race** ‚Äî `.github/workflows/deploy.yml` now makes `deploy-website-main` verify that `version.json` already has a matching published GitHub Release plus the expected CRX/installer assets before Cloudflare Pages can publish the new versioned website state
- **Static website release-surface drift** ‚Äî `/install` and `/downloads` now read the release version/repo directly from root `version.json` at build time, while `scripts/sync-constants.sh` generates `website/public/_redirects` and `website/public/updates.xml` from that same canonical source so Pages install/download/update surfaces cannot silently diverge during a hotfix
- **`scripts/check-version.sh` parser/website-hotfix false failures** ‚Äî The version guard now reads `website/src/pages/downloads.astro` from its real `currentVersion` constant, falls back to the legacy `textContent = 'v...'` pattern if present, reports `<missing>` mismatches instead of aborting on empty matches, validates that the install/download pages do not regress back to hardcoded website/API domains while the relative-origin/GitHub-release outage hotfix remains in place, and asserts both the release-tag wait on `create-release` and the master-release publish gate stay wired in CI
- **`scripts/check-version.sh` wrangler env-var dedupe regression** ‚Äî top-level `[vars]` plus `[env.production.vars]` now collapse identical quoted values during validation, while still failing when the same key resolves to divergent values across TOML scopes
- **Cloudflare Pages production branch targeting** ‚Äî `.github/workflows/deploy.yml` now passes `--branch=master` on both Pages deploy paths so CI deployments attach to the production Pages branch/custom-domain environment instead of creating detached deployments
- **Worker production environment vars** ‚Äî `workers/wrangler.toml` now duplicates required runtime values under `[env.production.vars]` because Wrangler does not inherit top-level `[vars]` into named environments
- **Cloudflare deployment docs** ‚Äî Root README plus setup guides now document the explicit Pages branch flag and the requirement to duplicate production Worker vars
- **Website install/download outage hotfix** ‚Äî The Astro install/download surfaces and Pages redirect shortcuts now bypass `autocr-api.nicx.me` for installer binaries by linking directly to GitHub Release assets derived from `version.json`, while website-hosted script downloads stay on relative `/scripts/...` paths and the install page‚Äôs copied one-liners resolve against the active website origin
